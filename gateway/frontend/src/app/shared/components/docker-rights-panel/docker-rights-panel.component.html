How does the Docker proxy work?

Every devcontainer gets its own Unix socket, mounted at /var/run/huddle; DOCKER_HOST points there. Docker commands therefore never go directly to the host's Docker daemon, but always pass through the Huddle proxy.

The proxy classifies each request into exactly one action (for example container.create or image.pull) and checks it against the toggles on this page; temporary actions additionally require an active timer. Unknown API paths are always rejected.

What Huddle injects or enforces along the way:

  • A huddle.parent label on everything the container creates (containers, images via build, volumes, and networks). This enforces ownership: listings are filtered on it and exec, logs, inspect, delete, and push only work on the container's own resources — never on devcontainers themselves.
  • Newly started containers are forced onto the container's own dc-net network and automatically receive the Huddle proxy environment variables and the CA certificate, so their internet traffic also passes through the egress firewall.
  • Dangerous settings are rejected: privileged containers, host namespaces (PID/IPC/UTS/userns/cgroup), extra capabilities, and disabled seccomp or AppArmor. Volume mounts are gated per kind under Volumes and off by default — host-path binds are the main escape vector, but named and anonymous volumes stay off until you enable them too. Binding host ports is only possible after approval in the port list.
  • docker volume prune is limited to the container's own volumes; docker push only works for self-built images and is off by default, because pushing bypasses the egress firewall.

Toggle changes and timer actions are recorded in the network log (audit).
